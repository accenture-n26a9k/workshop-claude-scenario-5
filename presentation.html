The Stage
Manager

A production-grade multi-agent intake system that triages, routes, and resolves operational requests from event crew in real time.

Built on the Claude Agent SDK  ·  Coordinator + 5 Specialists  ·  Deterministic safety brake

Situation
Every mid-size conference has one ops channel, one ops lead, and no triage.

"It's 10:47am on Day 2. The ops channel has 200 unread messages. One of them is a fire alarm."

The ops channel is not a public helpdesk. It is the internal Slack used by people with operational accountability — room captains, AV technicians, vendor contacts, sponsor liaisons, event staff.

These are the messages that carry real consequences. A missed message from a room captain means a speaker on a dead stage. A missed CoC report is a legal exposure. A missed safety page is a liability incident.

Who is in the channel

Room Captains

AV status, session start readiness

AV Technicians

Equipment failures, live session issues

Vendor Contacts

Catering, booth power, deliveries

Sponsor Liaisons

Escalations, SLA-bound requests

Event Staff

CoC reports, safety concerns

Key architectural constraint

No inherited context

Task subagents receive zero coordinator context. Every specialist operates only on what is explicitly passed in its Task prompt.

Validation-retry loop

Every classification passes through a Pydantic schema validator. On failure, the specific error is fed back to Claude and retried up to 3 times. After 3 failures → escalate with validation_failure=true.

Every decision is replayable

Each request logs: request_id, attempt_count, error_type, final_classification, routing_target, latency_ms.

The Brake System
Two independent layers. The first one is not an LLM decision. It cannot be prompted away.

Layer 1 — PreToolUse Hook (deterministic)

Before any write tool executes, a string match runs against a hardcoded keyword list. Not configurable via prompt. Not an LLM call.

SAFETY_KEYWORDS = [
  "fire", "alarm", "evacuation", "evac",
  "medical", "ambulance", "weapon", "bomb",
  "threat", "blood", "unconscious", "assault",
  "injury", "hurt", "emergency"
]

On match: safety lead paged · all agent action blocked · no auto-reply · no public log · written to safety audit trail only.

Layer 2 — Escalation rules (config, not prompt)
- condition: category == SAFETY
  action: HARD_PAGE  # always, no threshold
- condition: category == COC
  action: HUMAN_ONLY  # threshold: 0.4
- condition: category == PRESS
  action: HUMAN_ONLY  # always
- condition: confidence < 0.60
  action: ESCALATE  (non-FAQ)
- condition: dollar_impact > $5,000
  action: ESCALATE

Rules live in escalation_rules.yaml, not in a system prompt. Legal can audit without reading code. There is no prompt that changes these paths.

What We Deliberately Did Not Automate
Production systems are defined as much by what they refuse to do as by what they do.

Physical access control

The agent never opens doors, disables locks, or acts on physical security infrastructure — under any prompt, from any sender.

Code-of-Conduct responses

Never auto-replies to a CoC report. Never acknowledges receipt in a public channel. Never stores outside the isolated encrypted store.

Press / media statements

Any press inquiry routes to a human. The agent does not draft statements, confirm schedules, or provide data to press contacts.

Credential or permission changes

Cannot grant speaker access, change registration tiers, or modify crew permissions — regardless of instruction or apparent authority.

Financial commitments > $500

Any action implying spend authorization above $500 goes to human review. The agent does not make financial commitments.

Attendee-facing communication

The ops channel is for crew. Attendees are not in scope. The channel's value depends on staying that way.
